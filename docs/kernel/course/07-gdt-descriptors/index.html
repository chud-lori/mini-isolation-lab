Before Mini Kernel can switch privilege levels, it needs CPU descriptor tables. This lesson covers the host-tested x86_64 GDT entries used by early setup.

The Global Descriptor Table is old x86 machinery, but it still matters in long mode. A 64-bit kernel typically uses a small flat GDT with kernel code, kernel data, user code, and user data descriptors. Later, a task-state segment can be added for privilege transitions and interrupt stacks.

This milestone makes descriptor construction testable before the freestanding early-init path loads the table. The C helper builds exact eight-byte GDT entries, the default table layout, and a packed GDTR structure. Host tests assert the raw descriptor bits before `lgdt` uses them.

That split is deliberate. Descriptor bytes are easy to get subtly wrong, and a bad GDT can turn a boot bug into a reset or triple fault. Testing the bytes on the host gives the kernel path a stable foundation before segment reloads or ring-3 transitions.

Key Ideas

GDT entryEight bytes that describe a code, data, or system segment.
access bytePresent bit, privilege level, descriptor type, executable bit, and read/write bit.
flags nibbleLong-mode, 32-bit, and granularity flags packed next to the high limit bits.
GDTRThe CPU register payload containing the table base address and byte limit.
host testA way to prove descriptor bytes and selector order before `lgdt` loads them.

Source Reading Guide

Do not try to understand every instruction or helper in one pass. First trace the main path. Then return to the error path. Finally, connect the code back to the checkpoint.

  • Find how `kx86_gdt_entry_set` splits base, limit, access, and flags into bytes.
  • Find `kx86_gdt_default_table` and identify the null, kernel, and user descriptor order.
  • Open `tests/test_arch.c` and find the raw descriptor constants for kernel and user segments.

Checkpoint

Run `make check` and find the tests that prove the kernel code descriptor is exactly `0x00af9a000000ffff`.
