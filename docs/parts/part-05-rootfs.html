Part 05

Root Filesystem: What `/` Means

Show source: src/part-05-rootfs/chroot_exec.c
#define _GNU_SOURCE

#ifndef __linux__
#error "chroot_exec.c is intended for Linux."
#endif

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int main(int argc, char **argv) {
    if (argc < 3) {
        fprintf(stderr, "usage: sudo %s ROOTFS COMMAND [ARG...]\n", argv[0]);
        fprintf(stderr, "example: sudo %s ./rootfs /bin/sh\n", argv[0]);
        return 1;
    }

    if (chroot(argv[1]) < 0) {
        perror("chroot");
        return 1;
    }
    if (chdir("/") < 0) {
        perror("chdir");
        return 1;
    }

    execvp(argv[2], &argv[2]);
    perror("execvp");
    return 1;
}

A container needs more than process isolation. The process also needs a filesystem world. When a program opens `/bin/sh`, `/etc/passwd`, or `/lib/ld-linux`, those paths are resolved relative to a root filesystem.

`chroot` changes what a process sees as `/`. It is not a complete security boundary, but it is the simplest way to learn how a prepared root filesystem changes path lookup.

host / bin/ etc/ home/ container rootfs / bin/sh proc/ tmp/

What The Source Does

The program expects a rootfs path and a command. It calls `chroot(rootfs)`, then `chdir("/")`, then `execvp(command, args)`. After `execvp`, the setup program is gone and the requested command is running inside the new root view.

chroot(argv[1]);
chdir("/");
execvp(argv[2], &argv[2]);

Run It

make part-05-rootfs
mkdir -p rootfs/bin
cp /bin/busybox rootfs/bin/
ln -s busybox rootfs/bin/sh
sudo ./part-05-rootfs ./rootfs /bin/sh

Inside the shell, try `pwd`, `ls /`, and `echo hello`. Many normal commands will be missing unless your rootfs contains them. That is expected.

Why It Feels Broken

A rootfs is an environment. If it lacks `/proc`, process tools may fail. If it lacks shared libraries, dynamic binaries may fail. If it lacks DNS config, networking tools may fail. Container images exist because applications need more than a binary; they need a filesystem context.

Security Warning

`chroot` alone is not enough for untrusted code, especially if the process has high privileges. Real runtimes add mount namespaces, `pivot_root`, read-only mounts, user namespaces, seccomp, capability dropping, and LSM policy.

Checkpoint

You should be able to explain why `/bin/sh` inside a rootfs can be different from `/bin/sh` on the host, and why changing `/` is not the same as building a hardened container.
