Read-Only Filesystem: First Files

Before disk drivers exist, an in-memory read-only filesystem gives open, read, and stat real objects to operate on. It is the smallest useful file model for syscall tests.

Source focus: kernel/fs.c
All lessonsOpen source fileRead lesson

Article

Read this with kernel/fs.c open beside it. The goal is understanding the path, not memorizing every instruction or helper.

Before You Start

  • You understand strings, byte buffers, and small structs in C.
  • You know that files can be modeled as paths plus byte contents.

Terms Before Code

Filesystem

A naming and storage model for files and metadata.

Read-only

A filesystem mode where existing content can be read but not modified.

EOF

End of file: the point where no more bytes remain to read.

stat

A call that returns metadata such as size and mode.

Why Files Before Disk Drivers

A kernel can learn file semantics before it learns disk hardware. Mini Kernel uses an in-memory table of paths and byte slices. That is enough for open, read, close, and stat to have real behavior.

Descriptor State

When a file opens, the task gets a descriptor slot. Reads advance an offset. Closing releases the slot. This tiny lifecycle teaches the core idea behind POSIX file descriptors without needing a complete VFS.

Error Cases Are The Curriculum

The tests cover missing paths, EOF, bad descriptors, bad pointers, and stat errors. These cases matter because OS interfaces are contracts, and contracts are mostly about what happens when callers do the wrong thing.

What Comes Later

A future disk filesystem will need block devices, caching, directory formats, and write behavior. This lesson deliberately starts with the smallest model that lets syscalls become concrete.

Source Reading Checklist

  • Find the mounted file table.
  • Find how read advances an offset.
  • Find how stat fills the user-visible metadata struct.

Exercises

  • Add a new read-only file to the table and read it from init.
  • Explain what would be required to make write modify file contents.

Data Model

A file node is a path plus immutable bytes and metadata. An open descriptor points at a node and stores the current read offset. read copies bytes from that offset, advances the offset, and returns zero at EOF. stat reports metadata without consuming bytes.

Checkpoint

Run make all and find the tests that cover EOF, bad fd, missing paths, and stat errors.
