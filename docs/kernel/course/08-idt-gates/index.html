After descriptor segments come interrupt gates. This lesson builds host-tested x86_64 IDT entries for the table loaded by early setup.

The Interrupt Descriptor Table tells the CPU where to jump for exceptions, interrupts, and syscall-like trap entries. A missing or broken IDT makes early kernel faults hard to understand, because the CPU may reset instead of printing a useful diagnostic.

This milestone does not install handlers yet. It builds the packed 16-byte gate format, clears the default IDT table, builds the IDTR payload in C, and tests exact field placement on the host. That makes the early `lidt` step less mysterious and less fragile.

The helper supports interrupt gates and trap gates. Interrupt gates are the normal choice for hardware interrupts and faults because they clear the interrupt flag on entry. Trap gates leave it alone and are useful for controlled software-entry paths.

Key Ideas

IDT gateSixteen bytes containing a handler address, code selector, IST index, type, DPL, and present bit.
interrupt gateGate type 0xe, commonly used for faults and IRQs.
trap gateGate type 0xf, useful for controlled software traps.
IDTRThe CPU register payload containing the IDT base address and byte limit.
host testA way to prove handler address packing, privilege bits, and clear-table setup before loading the table.

Source Reading Guide

Do not try to understand every instruction or helper in one pass. First trace the main path. Then return to the error path. Finally, connect the code back to the checkpoint.

  • Find how `kx86_idt_gate_set` splits a 64-bit handler address into low, mid, and high fields.
  • Compare the interrupt-gate and trap-gate helpers and identify which type bits differ.
  • Open `tests/test_arch.c` and find the assertions for DPL bits, IST masking, and `kx86_idt_clear`.

Checkpoint

Run `make check` and find the test that proves an interrupt gate uses type attribute `0x8e`.
