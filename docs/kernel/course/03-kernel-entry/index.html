The C kernel path starts with serial logging because early kernels need a voice before they have files, terminals, processes, or rich drivers.

Freestanding C is not normal application C. You cannot assume printf, malloc, files, threads, an environment, or even that main is the shape of entry. The kernel has to bring its own tiny support routines and be careful about every dependency it introduces.

Serial logging is intentionally boring. That is the point. A simple serial output path gives the kernel a way to report progress and failure while everything else is still under construction. If a kernel cannot speak during early bring-up, debugging becomes guesswork.

The kernel_start path is designed to be readable. It initializes logging, prepares task state for syscalls, writes a banner through the syscall path, launches the built-in init program, checks its exit status, logs the result, and halts. That sequence is small enough to inspect line by line.

The key bridge is that this C path is currently developed and verified through freestanding objects and host-side tests. The BIOS image demonstrates the boot sector and stage2 monitor. A later loader step can connect those paths by loading the C kernel image and jumping into it.
