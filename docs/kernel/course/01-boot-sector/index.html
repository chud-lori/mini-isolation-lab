The BIOS boot path begins with a tiny and strict contract: one sector is loaded to address 0x7c00, and execution starts there. This lesson explains why that first binary is so constrained and why Mini Kernel immediately uses it to load stage2.

The first thing to unlearn is the idea that the machine loads an executable file for you. On this path, BIOS does not parse ELF sections, call a C runtime, or prepare argc and argv. It reads raw bytes from the boot disk into memory. The sector is considered bootable only if the final two bytes contain the boot signature 55 aa.

That is why the build cares about exact binary shape. The first sector must fit in 512 bytes, include the boot signature, and still contain enough code to prepare CPU state and read more sectors. If this grows accidentally, the image should fail validation instead of producing a mysterious non-booting artifact.

Real mode also explains why the assembly looks unusual. Segment registers such as CS, DS, ES, and SS participate in address formation. The stack must be placed somewhere predictable before calls and BIOS interrupts are safe. Disk access uses firmware services such as int 0x13 because the kernel has not written a disk driver yet.

Stage2 exists because the first sector should not become a crowded miniature OS. The boot sector preserves the boot drive, reads stage2, handles the obvious failure path, and jumps forward. That division gives you one tiny trusted loader and one roomier monitor for teaching and interaction.

Key Ideas

0x7c00The conventional physical address where BIOS loads the boot sector.
512 bytesThe size of the first sector BIOS loads.
55 aaThe required boot signature at the end of the sector.
int 0x13BIOS disk service used before Mini Kernel has its own driver.
