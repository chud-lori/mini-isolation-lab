Stage2 Monitor: A Visible Interface

Stage2 is still early real-mode code, but it makes the boot image feel alive. It gives you a tiny command monitor so you can observe the handoff instead of staring at a blank machine.

Source focus: boot/stage2.S
All lessonsOpen source fileRead article

Stage2 is still early real-mode code, but it makes the boot image feel alive. It gives you a tiny command monitor so you can observe the handoff instead of staring at a blank machine.

The monitor is not a full shell. It is a small loop: print a prompt, read characters, store them in a command buffer, compare the finished command against known strings, and dispatch the matching handler. That is enough to teach input, parsing, and control flow without hiding behind libraries.

Because stage2 still runs in the BIOS-era environment, it can lean on BIOS text and keyboard services. Keyboard input commonly comes through int 0x16, while text output commonly uses int 0x10. Those calls are convenient at this stage, but they are not permanent OS drivers. Later protected-mode or long-mode kernel code cannot simply depend on BIOS this way.

The command buffer is a tiny safety lesson. If input is stored into a fixed-size buffer, the code must decide what happens on Enter, Backspace, and too many characters. Even toy monitors need boundaries.

Stage2 also names the future handoff. Its text points toward kernel entry, syscall entry, userspace, and init. That does not mean the C kernel is already fully loaded by stage2 today. It means the monitor is a visible stepping stone in the course.
