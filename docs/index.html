1. The Mental Model

Isolation is about deciding what a program can see and what it can touch. A normal Linux process shares the host kernel with every other process. Containers add walls around that process. MicroVMs go further by putting a small virtual machine between the workload and the host kernel.

normal process: app -> host Linux kernel -> hardware container: app -> namespaces/cgroups -> host Linux kernel -> hardware microVM: app -> guest Linux kernel -> virtual hardware -> VMM/KVM -> host Linux kernel -> hardware

The most important difference is that a container shares the host kernel, while a microVM runs its own guest kernel.

3. Basic Linux Pieces

Process

A process is a running program. It has a process ID, memory, open files, environment variables, credentials, and kernel-managed resources.

Kernel

The kernel is the privileged part of the operating system. User programs do not directly control disks, network cards, memory mappings, or process scheduling. They ask the kernel.

Filesystem

Linux presents files, devices, and kernel views as paths. For example, `/proc` is not a normal directory on disk. It is a kernel-provided view of processes and system information.

Root Filesystem

The root filesystem is what a process sees under `/`. A container can be given a different root filesystem, so `/bin/sh` inside the container can be different from `/bin/sh` on the host.

4. What Is A Syscall?

A syscall is how a normal program asks the kernel to do privileged work. When you call `open`, `read`, `write`, `fork`, `clone`, `mount`, or `mmap`, your program eventually crosses from user mode into kernel mode.

printf("hello") -> C library -> write(1, "hello", 5) -> syscall instruction -> Linux kernel writes to terminal

Containers are built from Linux syscalls. The container demo uses `clone` to create a process with new namespaces, `chroot` to change the visible root filesystem, `mount` to mount `/proc`, and cgroup files under `/sys/fs/cgroup` to apply limits.

int flags = CLONE_NEWUTS | CLONE_NEWPID | CLONE_NEWNS |
            CLONE_NEWIPC | CLONE_NEWNET | SIGCHLD;

pid_t child_pid = clone(child_main, stack + STACK_SIZE, flags, &config);

That one `clone` call asks Linux to create a child process with new isolated views of hostname, process IDs, mounts, IPC, and networking.

5. Memory Allocation Basics

When a program asks for memory with `malloc`, it is asking the C runtime for memory. The C runtime may already have spare memory, or it may ask the kernel for more using syscalls such as `brk` or `mmap`.

malloc(1 MiB) -> C allocator -> maybe mmap() -> kernel maps virtual memory pages -> process receives an address it can use

Each process sees virtual addresses. The CPU and kernel translate those virtual addresses to physical memory. This is why two different processes can both have memory at address `0x1000` without touching the same real RAM.

The microVM demo uses `mmap` directly to allocate guest RAM inside the host VMM process:

uint8_t *guest_mem = mmap(NULL,
                          GUEST_MEM_SIZE,
                          PROT_READ | PROT_WRITE,
                          MAP_PRIVATE | MAP_ANONYMOUS | MAP_NORESERVE,
                          -1,
                          0);

Then it tells KVM: this block of host memory should appear as physical RAM to the guest VM.

6. How Containers Work

A container is not a tiny VM. It is a normal Linux process with isolated views and limits.

Feature Linux mechanism What it changes
Hostname isolation UTS namespace The process can have a different hostname.
PID isolation PID namespace The process can see itself as PID 1.
Filesystem view Mount namespace and rootfs The process sees a different `/` and mount table.
Network isolation Network namespace The process gets a separate network stack.
Resource limits cgroups The process can be limited by memory, CPU, and process count.
host Linux kernel process A process B mini-container process isolated hostname isolated PID view isolated mount view isolated network view cgroup limits

The container demo is intentionally small. It creates a child process with namespace flags, changes the root filesystem, mounts `/proc`, and runs your command.

Important: this demo teaches mechanics. It is not a hardened sandbox. Real container runtimes also drop capabilities, use seccomp, set up user namespaces, configure AppArmor or SELinux, manage images, and handle networking carefully.

7. How MicroVMs Work

A microVM is a small virtual machine optimized for fast startup, low overhead, and strong isolation. Firecracker is a production microVM monitor used for serverless and container workloads.

From the host, a Firecracker microVM is controlled by a host process. That process opens `/dev/kvm`, creates a VM, gives it memory, creates virtual CPUs, attaches virtual devices, and starts the guest.

host Linux firecracker process opens /dev/kvm creates VM maps guest RAM creates vCPU handles virtual disk/network/console guest Linux kernel guest app

The guest does not directly own your physical disk or network card. It sees virtual hardware. The host kernel still controls the real hardware.

The microVM demo is the smallest useful KVM loop:

open("/dev/kvm")
KVM_CREATE_VM
mmap guest memory
KVM_SET_USER_MEMORY_REGION
KVM_CREATE_VCPU
KVM_RUN
handle exits

When guest code writes to the fake debug port `0xe9`, KVM pauses the guest and returns to the VMM process with `KVM_EXIT_IO`. The VMM prints the byte, then resumes the guest.

guest writes byte to port 0xe9 -> CPU exits to KVM -> KVM returns to mini-firecracker -> mini-firecracker prints byte -> mini-firecracker calls KVM_RUN again

8. Why MicroVMs Are More Isolated Than Containers

The practical security difference is the kernel boundary.

Question Container MicroVM
Does workload share the host kernel? Yes. No. It has a guest kernel.
Where do app syscalls go? Directly to the host kernel. To the guest kernel first.
What does host expose? Large syscall surface. Virtual CPU, memory, and virtual devices.
Best for Trusted services, developer workflow, high density. Untrusted code, multi-tenant workloads, serverless.

A container escape often means abusing a host kernel feature or bug. A microVM escape usually requires crossing the guest kernel, virtual hardware, KVM, or VMM boundary. That extra boundary is why microVMs are attractive for running code from many customers on the same physical machine.

Firecracker makes this stronger by keeping the device model small. Fewer virtual devices means less code exposed to the guest.

9. Run The Demos

These demos require Linux. macOS does not provide Linux namespaces, cgroups, or KVM. Use a Linux VM, cloud instance, or Linux machine.

Build

sudo apt-get install build-essential busybox-static
make

Container Demo

mkdir -p rootfs/bin
cp /bin/busybox rootfs/bin/
ln -s busybox rootfs/bin/sh
ln -s busybox rootfs/bin/ps
ln -s busybox rootfs/bin/hostname
ln -s busybox rootfs/bin/mount
ln -s busybox rootfs/bin/ip

sudo ./mini-container ./rootfs /bin/sh

Inside the container:

hostname
echo $$
ps
mount

MicroVM Demo

ls -l /dev/kvm
./mini-firecracker

Expected output:

host VMM: starting vCPU
hello from a tiny microVM
host VMM: guest halted

10. Code Map

Container code

  • `src/part-07-container/mini_container.c`: creates namespaces with `clone`.
  • `try_apply_cgroup_limits`: writes to cgroup v2 control files.
  • `child_main`: sets hostname, changes root, mounts `/proc`, executes command.

MicroVM code

  • `src/part-09-microvm/mini_firecracker.c`: opens KVM and creates a small VM.
  • `guest_code`: tiny x86 machine code that writes characters to a fake device.
  • `handle_io_exit`: emulates the fake device when KVM exits to userspace.
