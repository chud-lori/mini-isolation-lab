Mini Kernel has a userspace-shaped init program today, but it is not pretending to have hardware user mode yet. This lesson explains why the simulation is still useful.

Real userspace means code runs with less privilege than the kernel, usually in ring-3 on x86. That requires page tables, privilege transitions, trap handlers, scheduler state, and a binary loading story. Mini Kernel is not all the way there yet.

What exists today is still meaningful. The init program has its own memory buffer, file descriptor slots, operation list, syscall frames, and exit status. It performs write, open, read, write, close, and exit through ksyscall_entry. That exercises the same kernel contract a real user program will need.

The useful flow is: kuser_op describes the next user action; the runner builds a trap-shaped frame; ksyscall_entry sends it to the dispatcher; the dispatcher validates and executes; the result returns to the user runtime. That is a small but honest model of an OS boundary.

Later, the operation list can be replaced with loaded user binaries, and the direct function call can be replaced with a CPU trap. The course keeps the current simulation visible so learners do not confuse “userspace-shaped” with “hardware-isolated.”
