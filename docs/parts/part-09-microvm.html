Part 09

Mini Firecracker: A Tiny VMM Event Loop

view source

Firecracker is a production virtual machine monitor for microVMs. This lesson does not recreate Firecracker. Instead, it builds the smallest useful VMM shape: create a VM, allocate guest memory, create a vCPU, run guest code, handle exits, and resume.

The Core Loop

while guest is alive:
  KVM_RUN
  inspect exit reason
  emulate device or event
  resume guest

While the guest runs ordinary CPU instructions, the host VMM is not interpreting every instruction. KVM and the CPU run guest code directly under virtualization controls. The VMM becomes active when KVM exits to userspace.

Guest Memory

The program allocates memory with `mmap`, copies tiny x86 guest code into it, and registers the region with KVM using `KVM_SET_USER_MEMORY_REGION`. The host sees process memory. The guest sees RAM.

The Fake Device

The guest writes bytes to I/O port `0xe9`. KVM exits with `KVM_EXIT_IO`. The VMM reads the byte from the shared `kvm_run` structure and prints it. That fake debug port is a tiny version of virtual device emulation.

case KVM_EXIT_IO:
    handle_io_exit(run);
    break;

Run It

make mini-firecracker
./mini-firecracker

Expected output:

host VMM: starting vCPU
hello from a tiny microVM
host VMM: guest halted

How This Maps To Firecracker

Real Firecracker boots a Linux kernel, attaches virtio block and network devices, exposes a local API, supports snapshots, emits metrics, and applies jailer/seccomp/cgroup restrictions. But the core idea is still recognizable: KVM runs guest CPU code, the VMM owns virtual machine configuration, and virtual device events are handled by userspace.

Why MicroVMs Are More Isolated Than Containers

A containerized app makes syscalls to the host kernel. A microVM app makes syscalls to the guest kernel. The host mostly sees virtual CPU, memory, and device interactions. That extra kernel boundary is why microVMs are attractive for untrusted or multi-tenant workloads.

Checkpoint

You should be able to explain `KVM_RUN`, VM exits, fake device emulation, and why a microVM is a VM even though Firecracker is just a host process.
