Mini Kernel

Mini Kernel is the course's tiny operating-system specimen: a bootable BIOS image, a second-stage monitor, and a freestanding C kernel scaffold. The point is not to pretend it is Linux. The point is to make the boot path, kernel boundary, memory model, and POSIX roadmap visible enough to learn from.

What You Build

The current source lives in mini-kernel/. It builds a raw image named build/mini-kernel.img. The image begins with a 512-byte BIOS boot sector, followed by a padded stage2 monitor. The C kernel code also compiles as freestanding x86_64 objects, ready for the later ELF bootloader path.

mini-kernel/
  boot/bootsector.S      BIOS entry at byte 0 of the disk image
  boot/stage2.S          tiny text monitor loaded by the boot sector
  boot/multiboot2.S      metadata for a future bootloader-loaded ELF kernel
  boot/linker.ld         section layout for that ELF kernel
  kernel/main.c          freestanding C kernel entry and serial logging
  kernel/arch/x86_64/    port I/O, serial, and halt helpers
  kernel/lib/            checked memory and string primitives
  tests/                 host-side tests for portable kernel logic

Boot, In Plain English

A normal program starts because an existing OS loader creates a process. This kernel starts before any process exists. On a BIOS-style machine, firmware reads the first 512 bytes of the disk into memory at physical address 0x7c00 and jumps there. Those 512 bytes are bootsector.bin.

The boot sector is tiny because BIOS only gives it one sector. Its job is to set predictable segment registers, remember the BIOS drive number, print a short status line, read stage2 from the next sectors into memory at 0800:0000, and jump there.

stage2.bin exists because serious setup does not fit comfortably in 512 bytes. Today it is a small monitor with commands like help, about, clear, halt, and reboot. Later it is the right place to set up protected mode or long mode, load the higher-level kernel, and pass boot information forward.

build/mini-kernel.img
  offset 0       bootsector.bin, exactly 512 bytes, ends in 55 aa
  offset 512     stage2.bin, padded to full 512-byte sectors

Why There Are Multiple Binaries

bootsector.bin is a raw 16-bit machine-code sector. BIOS does not understand C objects, ELF metadata, or Rust crates; it only checks that the first sector ends with the boot signature 55 aa and then executes it.

stage2.bin is also raw machine code, but it is loaded by our boot sector instead of by firmware. It gives the project room for real initialization.

mini-kernel.elf is the future bootloader-friendly kernel image. ELF is useful because it has sections such as text, rodata, data, and bss. That is the format a bootloader like GRUB or Limine can load cleanly once an ELF linker and emulator are available locally.

Run The Local Build

cd mini-kernel
make all

That runs host tests, builds the raw BIOS image, validates the 512-byte boot signature, validates stage2 sector padding, and compiles the freestanding kernel objects.

When qemu-system-x86_64 is installed, boot the raw image with:

cd mini-kernel
make run-image

POSIX Support Roadmap

You asked for POSIX support. That is the right ambition, but POSIX is not a switch; it is a stack of kernel features plus a C library ABI. Mini Kernel should grow toward a small POSIX-like subset in layers:

  1. Console and serial devices: enough for read and write.
  2. Interrupts, timer, and scheduler: enough for sleeping and process lifecycle.
  3. Virtual memory and user mode: separate kernel and process address spaces.
  4. Syscall ABI: stable entry from user programs into the kernel.
  5. File descriptors: stdin, stdout, stderr, pipes, and device handles.
  6. Process model: fork-like creation or a simpler spawn first, then exec and wait.
  7. Filesystem namespace: paths, directories, metadata, and a tiny initial RAM filesystem.
  8. libc surface: enough headers and wrappers for small POSIX-style programs.

The first compatibility target should be a tiny subset: write, read, exit, open, close, stat, getpid, and a simple spawn. Full POSIX means signals, terminals, permissions, time, sockets, and many edge cases; that comes much later.

Why C First, Not Rust Or Zig First?

The first implementation uses C and assembly because this machine already has a working no-network C/LLVM path for freestanding x86 objects and raw boot binaries. The available Rust install does not currently include the bare-metal core libraries or rustup target management needed for comfortable no_std x86_64 kernel work. Zig is not installed locally.

C is also useful pedagogically: it keeps the ABI, pointers, memory layout, and binary formats very visible. That is exactly what an OS course needs at the beginning. The tradeoff is that C gives fewer memory-safety guarantees, so this project keeps C narrow, uses checked helpers, treats hardware access as an explicit boundary, and tests portable logic on the host.

Rust is still a good future direction. Once the target libraries and linker path are available, Rust can take over larger kernel modules where ownership and bounds checks help. Zig could also be a strong fit because its cross-compilation story is good. The order is practical, not ideological: boot with what the local toolchain can build today, then introduce safer languages where they reduce risk.

Security Notes

The current OS is a teaching kernel, not a hardened system. The secure path is to add protections early: no writable-executable mappings, user/kernel address-space separation, checked syscall arguments, small driver interfaces, panic-on-invariant failure, and tests for allocator and page-table code before running untrusted programs.
