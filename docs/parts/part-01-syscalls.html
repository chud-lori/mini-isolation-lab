Part 01

Syscalls: Talking To The Kernel

Show source: src/part-01-syscalls/syscall_write.c
#define _GNU_SOURCE

#include <stdio.h>
#include <string.h>
#include <sys/syscall.h>
#include <unistd.h>

int main(void) {
    const char *message = "part 01: hello through the write syscall\n";

    /*
     * printf eventually calls write(2). Here we skip printf and ask the kernel
     * directly: write these bytes to file descriptor 1, which is stdout.
     */
    long ret = syscall(SYS_write, STDOUT_FILENO, message, strlen(message));
    if (ret < 0) {
        perror("syscall write");
        return 1;
    }

    return 0;
}

Before containers, KVM, Firecracker, or Docker make sense, you need one basic operating-system idea: a normal program does not control the machine directly. It runs in user mode. The Linux kernel runs in kernel mode. When a program needs privileged work done, it makes a syscall.

A syscall is a controlled entry point into the kernel. Printing text, opening files, creating processes, mapping memory, changing namespaces, mounting filesystems, and talking to KVM all use this boundary. The rest of this course is mostly about learning which syscalls or kernel interfaces are used to build isolation.

The Usual Path Is Hidden

When you call `printf`, your program usually does not jump straight into the kernel. `printf` formats text in userspace, then the C library eventually calls `write(2)` to send bytes to a file descriptor.

printf("hello")
  -> C library formats bytes
  -> write(1, "hello", 5)
  -> Linux kernel writes to stdout

File descriptor `1` is stdout. It may be your terminal, a file, a pipe, or a socket. The program does not need to know. This file-descriptor abstraction is one reason Unix tools compose so well.

What The Source Does

The source skips `printf` and calls the syscall interface directly:

long ret = syscall(SYS_write, STDOUT_FILENO, message, strlen(message));

The kernel receives a syscall number, a file descriptor, a pointer to bytes in process memory, and a length. It does not receive a friendly high-level string object. If the pointer is invalid, or the file descriptor cannot be written, the kernel rejects the call.

Run It

make part-01-syscalls
./part-01-syscalls
strace ./part-01-syscalls

With `strace`, look for `write(1, ...)`. You will also see loader and runtime syscalls. Ignore those for now. The important thing is seeing your program cross into the kernel.

Why This Matters For Isolation

A container runtime asks the kernel to create namespaces with `clone`. It asks the kernel to mount `/proc` with `mount`. It asks the kernel to change root with `chroot`. A microVM monitor asks KVM to create VMs with `ioctl`. These are different calls, but they all follow the same pattern: userspace asks, kernel validates, kernel performs or rejects.

Checkpoint

You should be able to explain why user programs need syscalls, why syscalls take raw arguments like pointers and lengths, and why every low-level runtime must check syscall return values.
