Containerization and microVMs

Learn isolation by building it in small parts.

This is a guided course for engineers without a low-level background. You start with one syscall, then memory, processes, namespaces, cgroups, and finally a tiny Firecracker-shaped KVM virtual machine monitor.

Before The First Part

The whole course is about one boundary: where a normal program stops and the kernel starts.

normal process: app -> host Linux kernel -> hardware container: app -> namespaces/cgroups -> host Linux kernel -> hardware microVM: app -> guest Linux kernel -> virtual hardware -> VMM/KVM -> host Linux kernel -> hardware

A container is a host process with isolated views. A microVM is a tiny VM with its own guest kernel. That is the security and architecture difference everything else builds on.

Part 01

Syscalls: talking to the kernel

source

A syscall is how a normal program asks the Linux kernel to do privileged work. Printing text, opening files, creating processes, mapping memory, and mounting filesystems all go through syscalls.

printf("hello")
  -> C library
  -> write(1, "hello", 5)
  -> Linux kernel
  -> terminal

The source for this part skips `printf` and calls `SYS_write` directly. That makes the user-to-kernel boundary visible.

Part 02

Memory: malloc, mmap, and virtual addresses

source

`malloc` asks the C allocator for memory. The allocator may ask the kernel for more pages using `mmap` or `brk`. A VMM also uses host memory, then tells KVM to expose that memory as guest physical RAM.

host VMM process
  mmap(2 MiB)
  -> host virtual memory
  -> KVM maps it as guest physical memory

This is why the microVM demo starts by allocating a memory region before it creates and runs a vCPU.

Part 03

Processes: clone without isolation

source

Before containers, you need processes. Linux has `fork`, but container runtimes commonly use `clone` because it can create a child process and choose which kernel resources are shared or isolated.

clone(child_main, stack + STACK_SIZE, SIGCHLD, NULL)

At this point there is no container yet. There is only a parent process and a child process.

Part 04

Namespaces: a different view of the same host

source

A namespace gives a process a different view of one system resource. The first namespace example uses `CLONE_NEWUTS`, which lets the child have a different hostname from the host.

clone(child_main, stack + STACK_SIZE, CLONE_NEWUTS | SIGCHLD, NULL)

Containers combine several namespaces: UTS for hostname, PID for process IDs, mount for filesystems, network for network interfaces, and IPC for inter-process communication.

Part 05

Root filesystem: what `/` means

source

`chroot` changes what a process sees as `/`. This is not a complete security boundary by itself, but it is a useful learning step toward root filesystems in containers.

host root: /bin /etc /home container rootfs: /bin/sh /proc /tmp

Real runtimes use stronger setup, often including `pivot_root`, bind mounts, read-only mounts, and careful capability dropping.

Part 06

Cgroups: limits, not views

source

Namespaces change what a process can see. Cgroups limit what a process can use. The example writes to cgroup v2 files like `pids.max`, `memory.max`, and `cgroup.procs`.

/sys/fs/cgroup/demo/
  pids.max       -> maximum number of processes
  memory.max     -> maximum memory
  cgroup.procs   -> PIDs that belong to this cgroup

This is the resource-control half of containers.

Part 07

Mini container: combine the pieces

source

The mini-container combines the earlier ideas: `clone` with namespace flags, a new root filesystem, a private mount namespace, `/proc`, and best-effort cgroup limits.

CLONE_NEWUTS | CLONE_NEWPID | CLONE_NEWNS |
CLONE_NEWIPC | CLONE_NEWNET
Learning tool, not hardened sandbox. Real runtimes also need user namespaces, seccomp, capability dropping, AppArmor or SELinux, image handling, and robust networking.
Part 08

KVM API: asking Linux to create VMs

source

KVM is the Linux kernel virtualization interface. A VMM opens `/dev/kvm`, sends ioctls, creates a VM, creates vCPUs, maps guest memory, and then repeatedly calls `KVM_RUN`.

open("/dev/kvm")
ioctl(KVM_CREATE_VM)
ioctl(KVM_CREATE_VCPU)
ioctl(KVM_RUN)

This part only checks the KVM API version. It is the smallest safe first touch before creating a full VM.

Part 09

Mini Firecracker: the VMM event loop

source

Firecracker is a production VMM. This course ends with a tiny VMM that shows the same core shape: create a VM, map guest memory, create a vCPU, run guest code, handle VM exits, and resume.

while guest is alive: KVM_RUN inspect exit reason emulate requested device or event resume guest

The guest writes characters to a fake I/O port. KVM exits to the host VMM, the VMM prints the character, and then the guest continues until it halts.

Container vs MicroVM Security

Question Container MicroVM
Does workload share host kernel? Yes. No. It has a guest kernel.
Where do app syscalls go? Directly to the host kernel. To the guest kernel first.
What does the host expose? Large syscall surface. Virtual CPU, memory, and devices.
Best fit Trusted services and developer workflow. Untrusted code and multi-tenant workloads.

Build And Try It

Linux required. macOS does not provide Linux namespaces, cgroups, or KVM. Use a Linux VM, Linux laptop, or cloud instance.

Build the course binaries

sudo apt-get install build-essential busybox-static
make
make parts

Run the mini-container

mkdir -p rootfs/bin
cp /bin/busybox rootfs/bin/
ln -s busybox rootfs/bin/sh
ln -s busybox rootfs/bin/ps
ln -s busybox rootfs/bin/hostname
ln -s busybox rootfs/bin/mount
ln -s busybox rootfs/bin/ip

sudo ./mini-container ./rootfs /bin/sh

Run the mini-firecracker

ls -l /dev/kvm
./mini-firecracker

Expected output:

host VMM: starting vCPU
hello from a tiny microVM
host VMM: guest halted

Mini Isolation Lab is educational code. It intentionally leaves out many production safety features. Do not treat either demo as a secure sandbox.
