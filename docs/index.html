Mini Isolation Lab

This is a step-by-step course for engineers who want to understand containerization and Firecracker-style microVMs from first principles. The site covers the repository's current fourteen Linux demo lessons, then adds a Mini Kernel companion for boot and kernel-boundary concepts.

How To Use This Course

Read the parts in order. Each lesson has its own page, source file, run commands, observable checkpoints, and notes about what the demo proves. The early parts are intentionally small because they introduce the kernel ideas used by the later runtimes.

The container side teaches how Linux isolates host processes with namespaces, root filesystems, and cgroups. The microVM side teaches how a host process uses KVM to run guest code behind a stronger VM boundary. The Mini Kernel side is separate: it shows what the machine looks like before Linux has created processes at all.

Kernel Companion

The Mini Kernel workbench and Mini Kernel course use the repository's tiny OS as learning material. Its working bootable path is a raw BIOS image: a 512-byte boot sector loads a stage2 text monitor. The freestanding C kernel, syscall dispatcher, read-only filesystem, userspace-shaped init path, and early x86_64 GDT/IDT setup are compiled and host-tested, while the Limine/Multiboot bootloader path is planned scaffolding.

Beginner To Expert Roadmap

The first fourteen Linux parts are the hands-on core. A complete isolation engineer needs more than that, so the course is organized as a roadmap from fundamentals to production-grade runtime design.

Module A: Beginner Foundations

  • Processes, file descriptors, and syscalls.
  • Virtual memory, page size, `malloc`, and `mmap`.
  • How to read `strace` output without getting lost.
  • Why return values and `errno` matter in systems code.

Module B: Container Internals

  • `clone` and process lifecycle.
  • UTS, PID, mount, IPC, network, and user namespaces.
  • Root filesystems, BusyBox, `chroot`, `pivot_root`, and `/proc`.
  • cgroup v2 controllers: memory, pids, CPU, and I/O.
  • Capability dropping, seccomp, AppArmor/SELinux, and why `root` inside a container is dangerous.

Module C: MicroVM Internals

  • KVM device model: `/dev/kvm`, VM file descriptors, vCPU file descriptors, and `ioctl`.
  • Guest memory layout and `KVM_SET_USER_MEMORY_REGION`.
  • vCPU state, real mode, protected mode, and long mode.
  • VM exits: I/O exits, halt exits, failure exits, and internal errors.
  • How virtio block and virtio net replace the toy debug port used in this lab.

Module D: Production Hardening

  • Threat model: container escape vs microVM escape.
  • Minimizing host attack surface with seccomp and a small device model.
  • Jailer-style process isolation around a VMM.
  • Rate limiting, metrics, logging, snapshots, and cleanup.
  • Operational lifecycle: create, start, stop, reap, delete, recover.

Module E: What To Build Next

  • Replace `chroot` with `pivot_root` in the mini-container.
  • Add a veth pair and basic NAT networking.
  • Boot a real Linux kernel in the mini-Firecracker path.
  • Implement a tiny Unix-socket control API inspired by Firecracker.
  • Add a minimal virtio-mmio device and compare it with the fake port device.
  • For Mini Kernel, link the C kernel into the BIOS stage2 handoff or make the planned Limine/Multiboot path the explicit bootloader route.

Run The Code

The demos are Linux-only. macOS does not expose Linux namespaces, cgroups, KVM, or Linux capabilities. `make` builds the combined container and microVM demos; `make parts` builds all numbered lesson binaries, including aliases for those combined demos.

sudo apt-get install build-essential busybox-static
make
make parts
