Part 02

Memory: malloc, mmap, And Virtual Addresses

Show source: src/part-02-memory/mmap_memory.c
#define _GNU_SOURCE

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/mman.h>
#include <unistd.h>

int main(void) {
    char *heap = malloc(64);
    if (!heap) {
        perror("malloc");
        return 1;
    }
    strcpy(heap, "malloc: memory managed by the C allocator");

    size_t page_size = (size_t)sysconf(_SC_PAGESIZE);
    char *mapped = mmap(NULL,
                        page_size,
                        PROT_READ | PROT_WRITE,
                        MAP_PRIVATE | MAP_ANONYMOUS,
                        -1,
                        0);
    if (mapped == MAP_FAILED) {
        perror("mmap");
        free(heap);
        return 1;
    }
    strcpy(mapped, "mmap: memory requested directly from the kernel");

    printf("%s\n", heap);
    printf("%s\n", mapped);
    printf("page size: %zu bytes\n", page_size);

    munmap(mapped, page_size);
    free(heap);
    return 0;
}

Every process sees virtual memory. The addresses in your program are not raw physical RAM addresses. The CPU and kernel cooperate through page tables to translate process-local virtual addresses into real memory pages.

This default memory isolation is one of the oldest and most important isolation layers. Process A cannot normally read Process B's memory because their virtual address spaces map to different pages.

malloc Is Not The Kernel API

`malloc` is a C allocator API. It manages chunks of memory for your program. Sometimes it can satisfy a request from memory it already owns. Sometimes it asks the kernel for more memory using `brk` or `mmap`.

`mmap` is closer to the kernel. It can create anonymous memory, map files, create shared mappings, or reserve virtual address ranges. The example uses anonymous private memory because that is the easiest mapping to understand.

char *mapped = mmap(NULL, page_size,
                    PROT_READ | PROT_WRITE,
                    MAP_PRIVATE | MAP_ANONYMOUS,
                    -1, 0);

Why VMMs Care About mmap

A microVM monitor allocates memory in the host process and then gives KVM a description of that memory. To the host, it is a virtual memory range owned by the VMM process. To the guest, KVM presents it as guest physical memory.

host VMM process mmap guest memory KVM_SET_USER_MEMORY_REGION guest sees that memory as RAM

This is why a microVM can have "its own memory" without directly owning the real memory controller. The host kernel remains in control of physical memory.

Run It

make part-02-memory
./part-02-memory

Expected output includes the allocator string, the mmap string, and the system page size. On many systems the page size is 4096 bytes, but code should ask instead of assuming.

Container vs MicroVM Memory

A container process uses host virtual memory directly. Cgroups can limit how much memory it uses, but there is no guest kernel memory layer. A microVM adds another level: guest virtual memory maps to guest physical memory, and guest physical memory maps to memory owned by the host VMM process.

Checkpoint

You should be able to explain the difference between `malloc` and `mmap`, why virtual addresses are process-local, and how host memory can become guest RAM.
