Part 03

Processes: clone Before Containers

view source

A process is a running program with its own address space, file descriptors, credentials, environment, and kernel bookkeeping. Containers do not replace processes. They start with processes and then add isolated views and limits.

Linux has `fork`, but this course uses `clone` because `clone` exposes the knobs that matter for containers. With different flags, `clone` can create something process-like, thread-like, or container-like.

The Smallest clone Shape

pid_t child = clone(child_main, stack + STACK_SIZE, SIGCHLD, NULL);

The child begins execution at `child_main`. The caller provides a stack because this form of `clone` starts the child at a function pointer. On common architectures the stack grows downward, so the example passes the high end of the allocated stack region.

What To Notice

`SIGCHLD` makes the child behave like a normal child process that the parent can wait for. The parent prints its PID, waits with `waitpid`, and frees the stack. The child prints its own PID and parent PID.

There is no namespace here. The child sees the same hostname, network stack, mount table, and process namespace as the parent. That is the point. You need to see plain process creation before adding isolation flags.

Run It

make part-03-processes
./part-03-processes

Expected output uses real PIDs from your machine:

parent: pid=1200 child=1201
child: pid=1201 parent=1200

Why This Matters

A runtime is responsible for lifecycle. It must start a process, know when setup has succeeded, handle failures, reap children, and clean resources. The final mini-container still does this. It just adds namespaces, cgroups, rootfs setup, and `/proc` mounting.

Checkpoint

You should be able to explain why `clone` needs a stack, what `SIGCHLD` does, and why this is not a container yet.
