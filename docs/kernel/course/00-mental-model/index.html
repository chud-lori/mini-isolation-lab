Primer

This page is for learners with basic C and terminal knowledge but little OS development background.

Normal App vs Kernel

A normal app starts after an existing OS creates a process, maps memory, provides files, connects stdout, and calls main. A kernel starts before those services exist. If the kernel wants files, tasks, memory protection, or output, it has to build the rules itself.

Firmware vs Boot Code vs Kernel

Firmware is the machine's first helper. In the BIOS path it loads the first 512-byte sector and jumps to it. Boot code is the tiny program that receives that jump and prepares the next stage. Kernel code is the OS core that eventually owns memory, devices, syscalls, and process rules.

Freestanding C

Freestanding C means the code is compiled without assuming a hosted OS runtime. You should not assume printf, malloc, files, environment variables, or threads exist. Mini Kernel keeps small helper libraries for checked memory/string work instead.

Two Paths Today

The current project has a BIOS boot image path and a host-tested C kernel path. The boot image runs the boot sector and stage2 monitor. The C kernel, syscall, filesystem, and userspace runtime compile and run through host-side tests today. A future lesson can connect the C kernel into the boot path as a loaded ELF kernel.

Why Syscalls Matter

A syscall is the controlled doorway from a user program into the kernel. Even before hardware ring-3 exists, Mini Kernel models the doorway with a trap-shaped frame, pointer validation, file descriptors, and return/error results.

Checkpoint

Explain the difference between a normal app, boot code, and kernel code. Then explain which Mini Kernel pieces run in the boot image today and which pieces are currently host-tested.
