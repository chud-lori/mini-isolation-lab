Part 04

Namespaces: A Different View Of The Same Kernel

Show source: src/part-04-namespaces/uts_namespace.c
#define _GNU_SOURCE

#ifndef __linux__
#error "uts_namespace.c uses Linux namespaces; build it on Linux."
#endif

#include <sched.h>
#include <signal.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/wait.h>
#include <unistd.h>

#define STACK_SIZE (1024 * 1024)

static int child_main(void *arg) {
    (void)arg;

    if (sethostname("part-04-namespace", strlen("part-04-namespace")) < 0) {
        perror("sethostname");
        return 1;
    }

    char hostname[128];
    if (gethostname(hostname, sizeof(hostname)) < 0) {
        perror("gethostname");
        return 1;
    }

    printf("child sees hostname: %s\n", hostname);
    return 0;
}

int main(void) {
    char hostname[128];
    if (gethostname(hostname, sizeof(hostname)) < 0) {
        perror("gethostname");
        return 1;
    }
    printf("parent sees hostname before clone: %s\n", hostname);

    char *stack = malloc(STACK_SIZE);
    if (!stack) {
        perror("malloc");
        return 1;
    }

    pid_t child = clone(child_main, stack + STACK_SIZE, CLONE_NEWUTS | SIGCHLD, NULL);
    if (child < 0) {
        perror("clone");
        free(stack);
        return 1;
    }

    waitpid(child, NULL, 0);

    if (gethostname(hostname, sizeof(hostname)) < 0) {
        perror("gethostname");
        free(stack);
        return 1;
    }
    printf("parent still sees hostname: %s\n", hostname);

    free(stack);
    return 0;
}

A namespace gives a process a scoped view of one kind of kernel resource. This is the most important container idea: the process is still running on the host kernel, but the kernel answers some questions differently for that process.

The first namespace in the course is UTS, which controls hostname and domain name. It is easy to observe and does not require building a filesystem or network setup.

The One-Flag Difference

clone(child_main, stack + STACK_SIZE, CLONE_NEWUTS | SIGCHLD, NULL)

The child calls `sethostname`, then reads the hostname back. The parent reads the hostname before and after the child runs. The child's hostname changes. The parent's hostname does not.

What This Proves

The host was not renamed. There is no second kernel. There is only one Linux kernel presenting different UTS namespace state to different processes.

Other namespaces use the same basic idea. PID namespaces change process visibility. Mount namespaces change mount-table visibility. Network namespaces change interfaces, routes, and sockets. IPC namespaces change certain inter-process communication resources.

Run It

make part-04-namespaces
sudo ./part-04-namespaces

Expected output:

parent sees hostname before clone: my-host
child sees hostname: part-04-namespace
parent still sees hostname: my-host

Container Connection

A container combines several namespaces. One namespace alone is not a useful container, but it gives you the pattern: isolate one view, verify the parent is unchanged, then add more isolated views until the process feels like it has its own small machine.

Checkpoint

You should be able to explain why namespaces are views, not VMs, and why changing hostname in the child does not rename the host.
