Build Check

The final lesson is not just a command list. It explains what the build proves, what it does not prove yet, and how to read common failures.

All lessonsRead article

A kernel course needs a feedback loop. For Mini Kernel, that loop is make all: build the raw BIOS image pieces, validate the boot sector and stage2 layout, compile the kernel objects, and run the host-side tests, including the x86_64 GDT and IDT descriptor setup.

What Success Means

A successful run means the BIOS assembly pieces build, the first sector remains valid, stage2 is sector-aligned, the raw image can be produced, freestanding descriptor-table setup compiles, and the portable C kernel logic passes tests. It is a strong signal that the boot-sector constraints, syscall dispatcher, filesystem model, syscall entry frame, descriptor layout, and userspace init path are internally consistent.

What Success Does Not Mean

It does not mean the C kernel is already boot-linked by stage2, and it does not mean hardware ring-3 isolation exists. It also does not mean useful fault handlers are installed yet or that the planned Limine/Multiboot ELF bootloader path is the active boot route. Disk drivers, networking, SMP, USB, UEFI, and complete POSIX compatibility are later system layers.

Commands

terminal
cd mini-kernel
make all

# when qemu-system-x86_64 is installed
make run-image

Troubleshooting

sector too largeThe boot sector grew beyond the BIOS first-sector contract.
bad signatureThe final boot marker is missing or not at the expected location.
host test failureLook at syscall, filesystem, userspace, or helper logic before blaming the boot image.
missing QEMUThe emulator is not installed. Host tests can still validate portable kernel logic.
