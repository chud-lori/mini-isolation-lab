A syscall is the controlled doorway from userspace into the kernel. Mini Kernel models that doorway with a trap-shaped frame now, so the validation and dispatch rules can be real before hardware ring transitions arrive.

User programs need kernel services. They want to write output, open files, read bytes, ask for process information, and exit. But they should not directly control kernel memory or hardware. A syscall is the narrow boundary where a request crosses into kernel code and gets checked.

Mini Kernel uses a register-shaped frame. In the common pattern, rax carries the syscall number, while registers such as rdi, rsi, and rdx carry arguments. The syscall entry function reads the frame, calls the dispatcher, and writes back a return value or error.

Consider write(fd, buf, len). The user operation places the syscall number in rax, the fd in rdi, the buffer pointer in rsi, and the length in rdx. The dispatcher then validates the task, checks that the buffer points inside allowed user memory, validates the descriptor, emits bytes, and returns either the number written or an error.

This is not hardware ring-3 yet. The current userspace runner calls ksyscall_entry directly. But the shape matters: a future trap handler can fill the same kind of frame from CPU state and reuse the dispatcher. That keeps today’s testable contract aligned with tomorrow’s hardware path.

Key Ideas

raxSyscall number.
rdiFirst argument, such as fd.
rsiSecond argument, such as buffer pointer.
rdxThird argument, such as byte length.
result/errorThe kernel response written back after dispatch.
