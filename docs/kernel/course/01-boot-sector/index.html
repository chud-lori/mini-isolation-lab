Boot Sector: The First 512 Bytes

The machine starts before there is an executable loader, process model, or C runtime. This lesson follows the BIOS contract and the exact reason the first Mini Kernel binary must fit into one sector.

Source focus: boot/bootsector.S
All lessonsOpen source fileRead lesson

Article

Read this with boot/bootsector.S open beside it. The goal is understanding the path, not memorizing every instruction or helper.

Before You Start

  • You can read simple assembly labels and jumps, even if every instruction is not familiar yet.
  • You know that memory addresses name byte locations.
  • You have seen hexadecimal numbers such as 0x7c00.

Terms Before Code

BIOS

Firmware code that starts the machine and knows how to load the first disk sector.

Boot sector

The first 512-byte sector of a BIOS boot disk. BIOS jumps into it if it ends with 55 aa.

Real mode

The early x86 CPU mode used by BIOS, with simple segmented addressing and no paging.

LBA

Logical Block Addressing: a way to refer to disk sectors by number.

The Starting Contract

A normal program starts because an existing operating system loads it. A kernel does not get that luxury. On the BIOS path, the firmware reads exactly one 512-byte sector from the selected boot disk into memory at physical address 0x7c00, then jumps there. That sector is just bytes. It is not an ELF file, not a Mach-O file, and not a C program with a runtime.

The Signature Rule

The final two bytes of the sector must be 55 aa. That signature tells BIOS the sector is bootable. If the signature is absent, firmware should not treat the sector as a boot program. This is why the build checks the binary size and signature instead of merely compiling assembly.

Why Stage2 Exists

The first sector is painfully small. It must set up enough CPU state, preserve the boot drive, print status, read more sectors, handle failure, and still fit in 512 bytes. Stage2 exists because the first sector should do only the minimum trustworthy handoff.

Reading The Source

Start at the entry label. Follow the code that sets segment registers, stores the boot drive, and invokes BIOS disk services. Do not try to memorize every instruction. Track three questions instead: where did execution begin, where will stage2 be loaded, and what happens on disk read failure?

Source Reading Checklist

  • Find the instruction that stores the boot drive number.
  • Find the disk read path that pulls in stage2 sectors.
  • Find the boot signature emitted at the end of the file.

Exercises

  • Change one message string in the boot path, rebuild, and confirm the image still passes the size/signature checks.
  • Explain why a boot sector cannot freely call printf or malloc.

Assembly Details To Notice

Segment registers such as CS, DS, ES, and SS describe how real-mode addresses are formed. The stack must be placed somewhere predictable before calls and interrupts can safely use it. BIOS interrupts such as int 0x13 are firmware services; the boot sector uses them because it does not have its own disk driver.

Expected Observation

After rebuilding, the image should still contain a 512-byte first sector ending in the boot signature. If the first-stage binary grows too large, the build should fail instead of silently producing a broken boot image.

Checkpoint

Explain why the first binary must be exactly 512 bytes and why the boot sector loads a second stage instead of becoming the whole kernel.
