Kernel Entry: Serial First

The freestanding C kernel path starts with the least glamorous but most valuable primitive: logging. Serial output gives the kernel a voice before higher-level devices exist.

Source focus: kernel/main.c
All lessonsOpen source fileRead lesson

Article

Read this with kernel/main.c open beside it. The goal is understanding the path, not memorizing every instruction or helper.

Before You Start

  • You know basic C functions, structs, and return values.
  • You understand that kernel C is freestanding: it cannot assume normal libc services exist.

Terms Before Code

Freestanding C

C compiled without assuming a hosted operating system environment or standard process runtime.

Serial port

A simple hardware text output path commonly used for early kernel logging.

Panic

A deliberate stop after detecting an unrecoverable kernel error.

Why Serial Comes First

Kernel code cannot assume stdout, files, terminal emulators, or a process runtime. Early serial logging is narrow, boring, and incredibly useful. It gives kernel code a way to report progress before the system has drivers, userspace, or a filesystem.

The Shape Of kernel_start

The kernel entry initializes logging, prepares task state for syscalls, writes a banner through the syscall path, launches the built-in init program, checks the exit status, logs the result, and halts. This makes the first kernel path a sequence you can read top to bottom.

Failure Is Part Of The Lesson

The code treats failed copies, failed writes, invalid task state, and failed init exits as explicit events. That is important in kernel work: a silent failure early in boot can look like nothing happened. Visible failure paths make the system teachable.

Why This Is Not Normal App C

In an app, main is called by an OS. In a kernel, the entry point is reached by boot code. There is no process, environment, argc, argv, filesystem, or allocator unless the kernel builds those pieces.

Source Reading Checklist

  • Find kernel_start and list its calls in order.
  • Find how the banner is written through the syscall path.
  • Find the halt path after init exits.

Exercises

  • Add one serial log line before launching init.
  • Explain which normal C library functions would be unsafe to assume here.

The Important Bridge

Do not assume stage2 already loads and jumps into this C kernel path. Today the BIOS image demonstrates boot and monitor mechanics, while the freestanding C kernel objects and host tests develop the kernel logic. The course keeps that boundary explicit so learners know what is implemented now and what still needs a loader handoff.

Checkpoint

Identify what happens if the banner copy or syscall write fails.
