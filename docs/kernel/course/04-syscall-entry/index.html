Syscall Entry: Trap-Shaped Boundary

The syscall boundary is shaped like a CPU trap even though it is still called directly by the current userspace runner. This lets the validation logic be real and testable today.

Source focus: kernel/syscall_entry.c
All lessonsOpen source fileRead lesson

Article

Read this with kernel/syscall_entry.c open beside it. The goal is understanding the path, not memorizing every instruction or helper.

Before You Start

  • You know that user programs should not directly control kernel memory.
  • You understand function arguments and return values in C.
  • You have seen CPU register names such as rax or rdi, even if only casually.

Terms Before Code

Syscall

A controlled request from userspace into the kernel.

Trap frame

A saved snapshot of CPU register-like state used to enter or resume privileged code.

Pointer validation

Checking that a user-provided address is inside memory the task is allowed to access.

File descriptor

A small integer handle a process uses to refer to an opened file or stream.

Why Syscalls Exist

User programs need services: write text, open files, read bytes, ask for process information, and exit. They should not directly poke kernel memory or hardware. A syscall is the narrow doorway where the kernel receives a request, checks it, and returns a result.

The Trap-Shaped Frame

Mini Kernel models a syscall entry frame with fields like rax, rdi, rsi, and rdx. rax carries the syscall number. The other registers carry arguments. The result and error fields are written back by the kernel. This shape is intentionally close to a future hardware trap path.

Dispatch And Validation

ksyscall_dispatch is where trust is reduced. It checks task pointers, user memory ranges, descriptor numbers, path lengths, and output buffers. This is the security lesson: never trust a syscall argument just because it arrived in a register.

What Is Real And What Is Future

Today, ksyscall_entry is called by the userspace runner, not by a real CPU privilege transition. But the dispatcher, task state, fd table, pointer checks, and error returns are real logic that can be tested now.

Source Reading Checklist

  • Find the frame fields used for syscall number and first three arguments.
  • Find the pointer range validation path.
  • Find one syscall that returns an error for a bad fd or bad pointer.

Exercises

  • Trace a write syscall from user operation to dispatcher result.
  • Add a test case for one invalid argument if it does not already exist.

Worked Example: write(fd, buf, len)

A userspace write operation places the syscall number in rax, the fd in rdi, the buffer pointer in rsi, and the length in rdx. ksyscall_entry converts that frame into a dispatch request. The dispatcher verifies the task, checks that the user buffer range is valid, validates the descriptor, writes bytes, and returns either a byte count or an error.

Checkpoint

Map rax, rdi, rsi, and rdx to syscall number and arguments for one write call.
